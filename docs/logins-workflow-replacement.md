# Replacing the Logins spreadsheet & mail-merge workflows with the IDM

This note maps the two manual onboarding workflows onto what the Identity Master
already does, and lays out what it would take to retire them.

- **Workflow A — the Logins spreadsheet.** HR keys new-hire / transfer data into
  Harris NextGen; we hand-copy it into a *Logins* workbook, and hand-copy the
  ALSDE ID from the [ALSDE Certification Search](https://tcert.alsde.edu/Portal/Public/Pages/SearchCerts.aspx).
- **Workflow B — the account-notification mail merge.** The Logins workbook is a
  Word mail-merge data source; we merge it into the *New Teacher* / *Non-Instructional*
  Technology Orientation Checklist and export to PDF (via the Adobe Acrobat plugin,
  because Word's own *Finish & Merge* breaks the embedded hyperlinks).

The short version: **the IDM already ingests almost every column of the Logins
spreadsheet automatically**, from the same two systems we copy by hand (NextGen +
PowerSchool). The spreadsheet is largely a manual re-keying of data the golden
record already holds. Three fields are genuine gaps. Once those are closed, the
spreadsheet can be replaced by an IDM export, and the mail merge can either point
at that export or be generated by the IDM directly.

## Workflow A — the Logins spreadsheet

### Column-by-column mapping

Every Logins column against where it already lives in the golden record. Sources:
`src/Import/FieldMap.php`, `src/Import/ColumnMap.php`, `docs/schema.sql`.

| Logins column | Already in IDM? | Golden record field | Feed source |
|---|---|---|---|
| Lastname | ✅ | `person.last_name` | NextGen `Last Name` |
| First Name MI | ✅ (first); ⚠️ (MI) | `person.first_name` + `person.middle_name` | First from NextGen; **middle name only comes from PowerSchool** — the NextGen ITExtract has no middle-name column (`ColumnMap`) |
| From School | ⚠️ derivable | prior `assignment` / `lifecycle_event` | not a first-class "from" field |
| From Position | ⚠️ derivable | prior `assignment.title` / `person.position_number` | not a first-class "from" field |
| To Position | ✅ | `person.position_number`, `assignment.title`, `assignment.job_code` | NextGen `Position Number` / `Job Code Desc` / `JOB CODE` |
| To School | ✅ | `person.primary_school_id` | NextGen `Location Code` |
| Effective Date | ✅ | `person.position_start_date` / `assignment.effective_date` | NextGen `Position Start Date` |
| End Date | ✅ | `person.end_date` / `assignment.end_date` | NextGen `Position End Date` |
| Board Approval | ❌ **gap** | — | not in any feed |
| Employee ID | ✅ | `person.employee_id` | NextGen `Employee Number` |
| DOB | ✅ | `person.dob` | PowerSchool `UsersCoreFields.dob` |
| Gender | ✅ | `person.gender` | NextGen `Gender Type` |
| Race | ✅ | `person.ethnicity_source` (+ resolved `ethnicity_code`) | NextGen `Ethnicity Description` |
| ALSDE ID | ✅ / ⚠️ new hires | `person.alsde_id` | PowerSchool `S_USR_X.state_staffnumber` |

**11 of 14 columns already flow in automatically** — no copy from NextGen needed.
The IDM even pulls NextGen and PowerSchool and *reconciles* them field-by-field
(the "Source field reconciliation" panel on each person page), so the data is not
just present but cross-checked.

### The three gaps

1. **ALSDE ID for brand-new hires.** The IDM already stores `alsde_id`, but its
   only source today is PowerSchool (`S_USR_X.state_staffnumber`). A person hired
   *before* they exist in PowerSchool has no ALSDE ID yet — which is exactly why
   the manual step scrapes the certification site. Note the schema already
   anticipates this: `person_source_id.system` includes an **`alsde`** enum value
   that nothing populates yet. Closing this means either an ALSDE lookup step, or
   accepting that ALSDE ID backfills automatically once the PowerSchool record
   appears (fine if the checklist doesn't need it on day one).

2. **Board Approval.** Not present in NextGen, PowerSchool, or any feed. This is a
   true net-new field — it needs somewhere to live (`person` or `assignment`
   column) and a way to enter it (the existing editor UI + RBAC), or a NextGen
   column we're not yet capturing if HR records it there.

3. **From School / From Position (transfer context).** The IDM stores the
   *current* assignment plus full history (`assignment` rows + `lifecycle_event`),
   so the prior school/position is *derivable*, but "from → to" isn't modeled as a
   first-class transfer. For a transfer we'd surface the previous primary
   assignment; for a brand-new hire these are simply blank.

### What replacing Workflow A looks like

Add a **Logins export / "new & changed staff" report** to the IDM, built from the
golden record:

- A filtered view — people who are `pending`/newly-activated, or whose assignment
  changed in a date window — rendered on screen and downloadable as CSV/XLSX.
- Columns = the Logins layout, populated from the fields above; "From
  School/Position" from prior assignment history; ALSDE ID as-available.
- Add the **Board Approval** field (editable, audited) and, if wanted, an **ALSDE
  lookup/entry** step so the one truly-manual value is captured once in the system
  of record instead of in a spreadsheet cell.

This removes the NextGen→spreadsheet hand-copy entirely and makes the data
auditable (every field change is already logged in `audit_log` /
`lifecycle_event`). It also removes the OneDrive-sync fragility the current
workflow depends on.

## Workflow B — the account-notification mail merge

Today the Logins workbook is the Word mail-merge data source; the merge fails on
hyperlinks unless the Acrobat plugin is used. The IDM makes two things possible:

1. **It knows the account outcome, not just the HR data.** Usernames are written
   back from OneSync and *locked* on the golden record (`person.username` /
   `email` / `upn`), and per-destination provisioning state (AD, Google, …) is
   tracked in `account_sync_status`, with live AD verification via Adaxes. The
   orientation checklist wants exactly this — who the person is *plus* their new
   username/email — which the spreadsheet never had.

2. **It can be the merge source, or do the merge.** Two options:

   - **Keep Word, swap the data source.** Point the existing `.docx` merge at an
     IDM export/view instead of the manually-maintained workbook. Lowest change;
     still uses Word (and still wants the Acrobat plugin for hyperlinks).
   - **Generate the checklist in the IDM.** Render the New Teacher /
     Non-Instructional checklist to PDF per person directly from the golden
     record (HTML→PDF template). This eliminates the Word/OneDrive/plugin chain
     and the *Finish & Merge* hyperlink bug altogether, and can be gated so a
     notice only generates once a username has been minted and locked.

The second option is the natural end state: the IDM already holds the person, the
assignment, and the minted account, so it can produce the notification as a
first-class artifact rather than exporting to a spreadsheet that Word then merges.

---

## Recommended sequence

1. **Add `board_approval`** (field + audited editor entry). Closes the only column
   with no possible feed source.
2. **Build the Logins export** (screen + CSV) from the golden record, including
   From/To assignment context. Retires the NextGen→spreadsheet copy.
3. **Decide the ALSDE story** — accept PowerSchool backfill, or add an ALSDE
   lookup/entry step (the `alsde` crosswalk system is already reserved).
4. **Replace the mail merge** — start by repointing the Word data source at the
   export; then move the checklist into the IDM as a generated PDF.

Steps 1–2 remove the bulk of the manual effort with the least new surface; 3–4
finish the job. None of it requires new data plumbing for 11 of the 14 columns —
the IDM is already pulling and reconciling them nightly.
